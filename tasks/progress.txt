## Codebase Patterns
- Claude Code hooks use `hookSpecificOutput` with `hookEventName` for decision control; PermissionRequest uses `decision.behavior` ("allow"/"deny"), PreToolUse uses `permissionDecision`
- Hook input JSON from Claude Code includes `hook_event_name`, `session_id`, `cwd`, `tool_name`, `tool_input` (object), `tool_use_id`, `transcript_path`, `permission_mode`
- PermissionRequest hooks do NOT receive `tool_use_id` (unlike PreToolUse) - must cache from PreToolUse if correlation is needed
- HookMessage struct in hook_models.rs uses snake_case fields; HookStatusPayload/PermissionRequestPayload use camelCase via serde rename
- The Tauri app uses `src-tauri/` for Rust backend, `src/` for React frontend, and `@tauri-apps/api` for IPC
- TypeScript types mirror Rust structs with camelCase field names
- Quality checks: `npx tsc --noEmit` for TS, `cargo check` for Rust (pre-existing objc macro warnings are expected)
- Claude Code settings.json hooks format: `{"hooks": {"EventType": [{"matcher": "", "hooks": [{"type": "command", "command": "python3 /path/to/script.py"}]}]}}`
- Hook installer uses `NOTCHAI_HOOK_MARKER = "notchai-hook.py"` to identify our entries for idempotent install/uninstall
- Codex CLI config is TOML at `~/.codex/config.toml`; notify uses `notify = ["command", "arg1"]` array format
- Codex hook installer uses `CODEX_NOTIFY_MARKER = "notchai-codex-notify.sh"` for idempotent install; script copied to `~/.notchai/hooks/`
- `codex_hooks_enabled` flag stored alongside `hooks_enabled` in `~/.notchai/config.json` (defaults to true)
- `map_hook_to_normalized_event()` now takes `agent` param; determines AgentType and EventSource dynamically (codex→Codex+Notify, default→Claude+Hook)
- Notchai config lives at `~/.notchai/config.json` with `hooks_enabled` boolean flag (default true)
- `install_hooks(source_script: &Path)` takes the resolved bundle resource path as input
- Hook server uses `tokio::sync::Mutex` (not `std::sync::Mutex`) for the pending map since it's held across `.await` points
- `useAgentMonitor` returns `pendingApprovals` and `respondToApproval` from `useHookEvents` — downstream components should use these for tool approval UI
- Hook status merging only applies to `claude` agent type sessions; event-bus merging applies to ALL agent types
- Status merge priority in useAgentMonitor: event-bus > hooks > poll (applied sequentially)
- Gemini CLI sessions are JSON files at `~/.gemini/tmp/<project_hash>/session-*.json` (respects `GEMINI_CLI_HOME` env var)
- New adapters need changes in 7 places: models.rs (AgentType enum + structs), adapters/<name>.rs (new), adapters/mod.rs, lib.rs (register), types.ts (AgentType), NotchArc.tsx (AGENT_TYPE_ORDER), pricing.ts
- `hook_server::get_server()` returns `Option<&'static HookServer>` for Tauri command access; `respond()` is `async`
- Socket server `start()` is `pub async fn` — must be spawned as a tokio task from Tauri setup
- Tauri 2 resource path resolution: `app.path().resolve("resources/notchai-hook.py", tauri::path::BaseDirectory::Resource)`
- Use `tauri::async_runtime::spawn()` to run async tasks from the synchronous setup block
- For app exit cleanup: use `.build().run(|_app, event| { ... })` pattern instead of `.run(context)` — allows handling `RunEvent::Exit`
- Tauri commands: `respond_to_approval` (async), `toggle_hooks_enabled`, `get_hooks_enabled`, `get_session_tool_calls` registered in invoke_handler
- Claude JSONL tool data lives in two places: `message.content[]` ToolUse blocks (assistant entries) and top-level `tool_result` entries — correlate by `tool_use_id`
- `AgentAdapter::get_tool_calls()` has default empty impl — only Claude and Gemini override it; Codex/Cursor use default
- `AgentMonitor::get_tool_calls()` tries each adapter and returns first non-empty result — session IDs are unique across adapters
- CSS `--island-expanded-height` and Rust `hover_height`/`expanded_height` must stay in sync; hover_height >= CSS height to keep mouse in zone
- Use `e.stopPropagation()` on nested interactive elements inside clickable session rows to prevent parent click handlers from firing
- OTEL server uses axum 0.8 with `Router::new().route(path, post(handler)).with_state(event_bus)` pattern; handler takes `(State<EventBus>, HeaderMap, Bytes)` extractors
- Hook event → NormalizedEvent mapping: PreToolUse→ToolStarted, PostToolUse→ToolCompleted, Stop/SubagentStop/SessionEnd→SessionEnded, SessionStart→SessionStarted, UserPromptSubmit→StatusChanged(Operating); PermissionRequest stays on direct Tauri path
- EventBus Tauri bridge pattern: `event_bus.subscribe()` returns a broadcast::Receiver, loop with `rx.recv().await` and emit Tauri events; handle `Lagged` by logging and continuing, `Closed` by breaking
- opentelemetry-proto 0.28 with features `["gen-tonic-messages", "trace"]` provides protobuf message types at `opentelemetry_proto::tonic::collector::trace::v1::*`
- OTEL span attribute values use `opentelemetry_proto::tonic::common::v1::any_value::Value` enum — extract string with `Value::StringValue(s)`
- For graceful port-already-in-use: `match tokio::net::TcpListener::bind(addr).await { Err(e) => { log warning; return; } }` — app keeps running without OTEL server
